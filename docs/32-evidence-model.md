# VALOUR Evidence Model

Version: 0.1
Status: Private evidence model
Release: Pilot Ready

## Purpose

This document defines how VALOUR captures useful evidence during the pilot.

The evidence model is designed to show whether the pilot helps real users prepare for and handle real architecture leadership conversations better.

## Evidence principle

Useful evidence is not polite interest.

Useful evidence shows a change in preparation, clarity, confidence, behaviour, outcome, or willingness to continue.

## Evidence types

### Baseline evidence

Captured before VALOUR support.

Fields:

- User role
- Scenario type
- Confidence before
- Main concern
- Desired outcome
- Expected difficulty

### Preparation evidence

Captured during preparation.

Fields:

- Situation frame
- Decision required
- Audience needs
- Key message
- Likely questions
- Risks to avoid
- Closing statement

### Rehearsal evidence

Captured during rehearsal.

Fields:

- Prompt used
- User response
- Feedback
- Improved language
- Pattern observed

### Real-world evidence

Captured after the real conversation or meeting.

Fields:

- What happened
- Outcome
- Stakeholder response
- User confidence after
- What preparation helped
- What was still difficult

### Improvement evidence

Captured after review.

Fields:

- Main improvement
- Main gap
- Next development focus
- Whether the user would repeat the process
- Whether the user would recommend it
- Whether the user would support a next step

## Evidence quality levels

Weak signal: the user says the idea is interesting but takes no action.

Moderate signal: the user completes a rehearsal and says it was useful.

Strong signal: the user applies the preparation in a real situation and reports a clear improvement.

Very strong signal: the user wants to continue, recommends another user, supports a next step, or introduces a team buyer.

## Evidence review cadence

Review evidence after every pilot user.

After five users, review patterns across all evidence and decide whether to continue, refine, price, or build.

## Evidence risks

Do not mistake encouragement for validation.

Do not rely only on compliments.

Do not ignore negative feedback.

Do not collect sensitive organisational detail unnecessarily.

Do not claim outcomes that were not observed.

## Decision gate

Proceed to product build only when evidence shows that users complete the loop, report usefulness, and show willingness to continue, recommend, support a next step, or introduce others.
