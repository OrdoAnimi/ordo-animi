# VALOUR MVP and MLP Definition

Version: 0.1
Status: Private product definition
Phase: Architecture and Pilot Build

## Purpose

This document defines the Minimum Viable Product and Minimum Lovable Product for VALOUR for Architecture Leadership.

The MVP proves the workflow. The MLP creates an experience that early users would want to repeat and recommend.

## Product principle

MVP proves usefulness.

MLP creates trust, clarity, and repeat desire.

VALOUR should not become platform-heavy before the prepare, rehearse, review, and improve loop is proven.

## MVP definition

The MVP is the smallest product experience that allows one architecture leader to complete the VALOUR loop end to end.

The MVP must support:

- User intake
- Scenario selection
- Preparation brief
- Rehearsal prompt
- Language refinement
- After-action review
- Pattern report
- Evidence capture

## MVP success criteria

The MVP is successful if a user can bring one real architecture leadership situation and complete the loop with useful outputs.

The user should leave with:

- A clearer understanding of the situation
- A prepared message
- Better language options
- Improved confidence
- A completed review
- A pattern insight
- A clear next improvement focus

## MVP release shape

The first MVP can be manual or lightly assisted.

It can run using:

- Markdown templates
- Private GitHub repository
- Prompt pack
- Manual AI execution
- Manual pattern report
- Manual evidence tracker

## MVP exclusions

The MVP does not include:

- Full SaaS platform
- Enterprise login
- Payment
- Live meeting assistant
- Calendar integration
- Mobile app
- Full VITA product
- Full desktop companion
- Automated multi-agent orchestration

## MLP definition

The Minimum Lovable Product is the smallest version that feels valuable, coherent, and worth repeating.

The MLP should feel like VALOUR understands the architecture leadership context and helps the user walk into the real meeting better prepared.

## MLP experience goals

The user should feel:

- Clearer
- Calmer
- Better prepared
- Less likely to over-explain
- Better able to handle questions
- More deliberate with language
- More aware of their leadership pattern

## MLP features

The first MLP should add:

- Polished intake experience
- Guided scenario selection
- Clear preparation output
- Stronger rehearsal flow
- Better language refinement
- Clean pattern report
- Repeatable pilot summary
- Simple visual identity
- User-friendly landing page

## MVP versus MLP

| Area | MVP | MLP |
|---|---|---|
| Intake | Manual template | Simple guided form |
| Scenario selection | Operator selected | Guided scenario choice |
| Preparation | Prompt generated | Polished briefing format |
| Rehearsal | Manual prompt | Guided rehearsal experience |
| Report | Manual notes | Clean pattern report |
| Evidence | Tracker | Simple dashboard or summary |
| Experience | Useful | Repeatable and credible |

## Release mapping

### Release 1: Manual MVP

Goal: prove the loop with one user.

### Release 2: Repeatable MVP

Goal: make the pilot repeatable across several users.

### Release 3: MLP Pilot Experience

Goal: make the experience polished enough to share with advisors, early users, and team buyers.

### Release 4: Digital MVP

Goal: introduce a lightweight interface.

### Release 5: Agentic MLP

Goal: introduce structured agent workflow and stronger reporting.

## Decision gate

Move from MVP to MLP only after the core loop proves useful.

Move from MLP to platform only after users show repeat intent, recommendation signal, or team interest.
