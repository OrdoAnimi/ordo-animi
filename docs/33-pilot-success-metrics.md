# VALOUR Pilot Success Metrics

Version: 0.1
Status: Private metrics model
Release: Pilot Ready

## Purpose

This document defines how the VALOUR pilot will judge success.

The pilot is not trying to prove a full platform yet. It is trying to prove that the core behaviour loop is useful.

## Core success question

Did VALOUR help the user prepare for, handle, and learn from a real architecture leadership conversation better than they would have without it?

## Primary metrics

### Pilot completion

A pilot is complete when the user finishes intake, scenario selection, preparation, rehearsal, real-world use, after-action review, and pattern report.

Target: at least 80 percent completion across pilot users.

### User usefulness score

The user rates how useful VALOUR was after the pilot.

Scale: 1 to 10.

Target: average score of 7 or higher.

### Confidence shift

The user rates confidence before and after VALOUR support.

Target: average positive shift of at least 2 points.

### Scenario relevance

The user rates whether the selected scenario matched their real situation.

Scale: 1 to 10.

Target: average score of 8 or higher.

### Repeat intent

The user indicates whether they would use VALOUR again for another leadership situation.

Target: at least 60 percent yes.

## Secondary metrics

### Recommendation signal

Would the user recommend VALOUR to another architect or technical leader?

Target: at least 50 percent yes after early pilot.

### Sponsorship signal

Would the user, manager, practice lead, or organisation support further use?

Target: at least one clear sponsorship or team-pilot signal within the first pilot set.

### Scenario expansion signal

Did the user identify additional scenarios that should be added?

Target: capture recurring scenario gaps after five users.

### Language improvement signal

Did the user report that the language refinement helped them speak more clearly, more concisely, or with better tone?

Target: majority of users report at least one useful language improvement.

### Trust signal

Did the user report improved stakeholder response, clearer decision framing, or reduced friction?

Target: at least two concrete examples from the first pilot set.

## Qualitative success signals

Strong qualitative signals include:

- The user says they were clearer.
- The user says they handled questions better.
- The user says they avoided over-explaining.
- The user says they were calmer.
- The user says the conversation went better than expected.
- The user asks to use VALOUR again.
- The user suggests another person who needs it.
- The user asks whether a team version exists.

## Failure signals

Failure signals include:

- The user cannot connect VALOUR to a real situation.
- The scenario feels artificial.
- The prompts are too generic.
- The output is too long to use.
- The user does not complete real-world application.
- The user says the process was interesting but not useful.
- The user would not use it again.

## First decision gate

After five pilot users, decide one of four paths:

1. Continue architecture leadership pilot unchanged.
2. Refine scenarios and run another pilot set.
3. Narrow to the highest-value scenario.
4. Pause and redesign the offer.

## Build decision gate

Do not build a full platform until there is evidence that users complete the loop, find it useful, and want to continue.

A lightweight product build can begin when there is strong evidence across completion, usefulness, repeat intent, and sponsorship signal.
