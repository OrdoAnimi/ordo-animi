# VALOUR Execution Sprint Plan

Version: 0.1
Status: Private execution plan
Phase: Execution Sprint

## Purpose

This document moves the project from early discovery into execution mode.

The founder has already had prior founder and investor support, sprint setup, meetings, and product discussion around the original VALOUR concept. A related version appears to have been redirected into a different vertical by others. That is treated as signal, not as a blocker.

The next move is to push forward with the founder's strongest vertical: architecture leadership.

## Execution position

We are not starting from zero validation.

There has already been enough belief in the core concept to justify product sprinting.

The immediate task is not to prove that AI-assisted leadership support is theoretically interesting. The task is to package and execute VALOUR in the architecture vertical.

## Strategic correction

The first product is not generic leadership.

The first product is not a different industry vertical.

The first product is not wellness.

The first product is VALOUR for Architecture Leadership.

This returns the product to the founder's original vertical and strongest credibility base.

## Sprint objective

Create a pilot-ready, founder-led VALOUR package that can be used with architecture leaders without further conceptual explanation.

## Sprint 1: Pilot package hardening

Goal: make the pilot usable with the first external user.

Deliverables:

- Pilot pack reviewed and tightened.
- Intake form finalised.
- Scenario prompt pack refined.
- Discovery call notes ready.
- Validation tracker ready.
- Landing page draft ready for future site build.

Definition of done:

A first user can be approached, onboarded, prepared, rehearsed, and reviewed using the current documentation.

## Sprint 2: First user simulation

Goal: run VALOUR internally against a realistic architecture scenario before using it externally.

Deliverables:

- Create one sample architecture leadership scenario.
- Complete the intake form as a test user.
- Run preparation prompt.
- Run rehearsal prompt.
- Run language refinement.
- Run after-action review.
- Capture gaps in the pilot flow.

Definition of done:

The manual VALOUR loop works end to end in a controlled test.

## Sprint 3: First contact list

Goal: identify the first five real contacts.

Deliverables:

- Contact list with role, relationship, likely pain, and approach angle.
- Selected outreach message for each contact.
- Tracker rows prepared.
- Priority order agreed.

Definition of done:

Five contacts are ready for outreach.

## Sprint 4: First external conversations

Goal: move from internal readiness to external signal.

Deliverables:

- Send outreach to first contacts.
- Run discovery calls where accepted.
- Capture pain language.
- Identify best scenarios.
- Identify pilot candidates.
- Identify advisor or funder follow-up.

Definition of done:

At least three meaningful external conversations completed or scheduled.

## Sprint 5: Pilot execution

Goal: run the first real VALOUR pilot.

Deliverables:

- Select first user or small team.
- Complete intake.
- Select scenario.
- Run preparation.
- Run rehearsal.
- User applies in real situation.
- Run after-action review.
- Capture feedback and improvement.

Definition of done:

One real VALOUR pilot cycle completed.

## Delivery principles

Keep the product narrow.

Use architecture leadership language.

Do not over-explain Ordo Animi.

Do not chase another vertical.

Do not wait for perfect software.

Do not build the full desktop agent yet.

Prove the manual behaviour loop first.

## Immediate next action

Create a simulated pilot case and run the VALOUR process internally.

This will expose whether the pilot pack is operational or still only conceptual.
