# Ordo Animi™ Product Roadmap

Version: 0.1
Status: Private roadmap

## Purpose

This document defines the staged roadmap for Ordo Animi™ and VALOUR™.

The roadmap separates the long-term agentic operating system vision from the first buildable and sellable product.

## Product vision

Ordo Animi is an agentic life-and-work operating system.

It helps a person move between professional mode, personal mode, truth-checking, recovery, and long-term development without losing context or command.

The long-term vision is a desktop and mobile companion that behaves like a contextual operating system. It understands the user's workday, meetings, pressure points, personal commitments, recovery needs, and mode transitions.

## First product

The first commercial product is VALOUR for Architecture Leadership.

VALOUR helps architects and technical leaders prepare for, rehearse, and review high-stakes architecture leadership conversations.

## Roadmap principle

Do not build the full platform first.

Prove the behaviour loop first.

The product should move from manual protocol to repeatable pilot, then to lightweight tooling, then to integrated agentic experience.

## Phase 0: Founder workspace

Status: in progress.

Goal: create the private product source of truth.

Outputs:

- Product charter
- Founder story
- System architecture
- Market positioning
- Go-to-market strategy
- Funder engagement pack
- MVP plan
- Scenario library
- Pilot templates
- Outreach copy
- Pitch narrative

Success criteria:

The concept is contained, documented, and ready for pilot planning.

## Phase 1: Manual VALOUR pilot

Goal: test whether real users get value from scenario-based AI rehearsal for architecture leadership.

Inputs:

- User intake
- Real upcoming leadership scenario
- Scenario preparation template
- AI rehearsal prompt
- After-action review
- Pattern snapshot

Outputs:

- Individual user feedback
- Scenario improvements
- Evidence of value
- Testimonials where appropriate
- Refined pilot offer

Success criteria:

Five to ten users or one small architecture team completes the process and reports meaningful value.

## Phase 2: Repeatable pilot product

Goal: turn the manual pilot into a repeatable service product.

Outputs:

- Defined four-week pilot pack
- Reusable scenario prompts
- Standardised intake form
- Standardised feedback form
- Pilot completion report template
- Basic landing page
- Clear pricing ladder

Success criteria:

The pilot can be delivered repeatedly without reinventing the process each time.

## Phase 3: Lightweight digital MVP

Goal: create a simple interface for the core VALOUR loop.

Core features:

- Scenario selection
- Meeting preparation flow
- AI rehearsal entry point
- Language refinement
- After-action review
- Pattern history
- Exportable notes

Success criteria:

The user can complete the prepare, rehearse, review loop without founder handholding.

## Phase 4: VALLUM handover layer

Goal: introduce mode switching between professional and personal state.

Features:

- End-of-day professional closure
- Open-loop capture
- Emotional residue check
- Handover from VALOUR to VITA
- Personal evening prompt

Success criteria:

The user can close work mode and enter personal mode with less carryover.

## Phase 5: VITA personal layer

Goal: introduce the personal leadership and recovery companion.

Features:

- Recovery prompt
- Health and movement reminders
- Social connection checks
- Family and relationship prompts
- Shutdown routine
- Personal reflection

Success criteria:

The user experiences VITA as a personal support layer, not as generic wellness content.

## Phase 6: Integrated agentic operating system

Goal: build toward the long-term Ordo Animi vision.

Possible integrations:

- Calendar
- Tasks
- Notes
- Meeting summaries
- Daily logs
- Personal routines
- Work context
- User memory

Success criteria:

The system can brief the user in the morning, prepare specific leadership moments during the day, close work mode, and hand over to personal mode.

## Phase 7: Sector expansion

Goal: expand beyond architecture leadership only after proof.

Possible expansion areas:

- Technical leadership
- Delivery leadership
- Consulting leadership
- First-time managers
- Small business owners
- Industry-specific difficult conversation products
- Personal operating systems

## Roadmap risk

The main risk is overbuilding before validation.

The full Jarvis-style operating system is the vision, not the MVP.

The first build must remain VALOUR for Architecture Leadership.

## Roadmap discipline

Every phase must answer one question:

Does this help the user prepare for, handle, or learn from a real leadership moment better than before?

If not, it waits.
