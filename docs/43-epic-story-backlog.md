# VALOUR Epic and Story Backlog

Version: 0.1
Status: Private backlog
Phase: Architecture and Pilot Build

## Purpose

This document defines the first epic and story backlog for VALOUR for Architecture Leadership.

The backlog translates the architecture and pilot workflow into deliverable work items.

## Backlog principle

Every story must support the VALOUR loop:

Prepare.

Rehearse.

Refine.

Apply.

Review.

Report.

Learn.

## Epic 1: Pilot Foundation

### Goal

Create the minimum operating foundation for the manual pilot.

### Story 1.1: Pilot folder structure

As a pilot operator, I want a standard folder structure so that each pilot user can be managed consistently.

Acceptance criteria:

- Standard folders exist for users, scenarios, rehearsals, reviews, and reports.
- Each folder has a README or template.
- The structure supports one user end to end.

### Story 1.2: Pilot record templates

As a pilot operator, I want reusable record templates so that I do not recreate notes for each user.

Acceptance criteria:

- Intake template exists.
- Scenario template exists.
- Preparation template exists.
- Rehearsal template exists.
- Review template exists.
- Pattern report template exists.

### Story 1.3: Pilot runbook

As a pilot operator, I want a step-by-step runbook so that I can run the pilot consistently.

Acceptance criteria:

- Runbook covers intake to pattern report.
- Runbook includes operator actions.
- Runbook includes completion criteria.

## Epic 2: Scenario and Prompt Workflow

### Goal

Make the scenario catalogue usable in a pilot setting.

### Story 2.1: Scenario catalogue refinement

As a pilot user, I want scenarios that match real architecture leadership situations so that the pilot feels relevant.

Acceptance criteria:

- Core architecture leadership scenarios are documented.
- Each scenario has a clear situation and outcome.
- Scenarios are not overly generic.

### Story 2.2: Scenario selection guide

As a pilot operator, I want a scenario selection guide so that user situations can be mapped quickly.

Acceptance criteria:

- User intake maps to scenario options.
- Each scenario has selection criteria.
- Operator can identify a variant if needed.

### Story 2.3: Scenario prompt pack

As a pilot operator, I want reusable prompts so that preparation, rehearsal, refinement, and review can be run consistently.

Acceptance criteria:

- Preparation prompt exists.
- Rehearsal prompt exists.
- Language refinement prompt exists.
- Review prompt exists.

## Epic 3: User Experience

### Goal

Make the pilot understandable and usable for the first participant.

### Story 3.1: Pilot pack

As a pilot user, I want a simple explanation of the pilot so that I understand what I am joining.

Acceptance criteria:

- Pilot pack explains what VALOUR is.
- Pilot pack explains what the user brings.
- Pilot pack explains what the user receives.

### Story 3.2: Intake form

As a pilot user, I want a clear intake form so that I can explain my situation without writing an essay.

Acceptance criteria:

- Intake form captures role, context, scenario, confidence, and desired outcome.
- Intake form avoids unnecessary sensitive information.
- Intake form is easy to complete.

### Story 3.3: Pattern report

As a pilot user, I want a concise pattern report so that I understand what improved and what to work on next.

Acceptance criteria:

- Report includes scenario, starting challenge, improvement, pattern, and next focus.
- Report is short enough to read quickly.
- Report is useful without requiring explanation of Ordo Animi.

## Epic 4: Evidence and Validation

### Goal

Capture meaningful pilot evidence.

### Story 4.1: Evidence model

As a founder, I want evidence categories so that I can judge whether the pilot is working.

Acceptance criteria:

- Baseline evidence is defined.
- Preparation evidence is defined.
- Rehearsal evidence is defined.
- Real-world evidence is defined.
- Improvement evidence is defined.

### Story 4.2: Success metrics

As a founder, I want clear pilot metrics so that I can decide whether to continue, refine, or build.

Acceptance criteria:

- Completion metric exists.
- Usefulness metric exists.
- Confidence shift metric exists.
- Repeat intent metric exists.
- Recommendation signal exists.

### Story 4.3: Validation tracker

As a founder, I want a tracker so that I can record contacts, calls, pilot candidates, evidence, and next actions.

Acceptance criteria:

- Tracker exists.
- Status values are defined.
- Decision gate is defined.

## Epic 5: Architecture and Platform

### Goal

Prepare for the first implementation decision without overbuilding.

### Story 5.1: Architecture input brief

As an architect, I want functional and non-functional requirements so that platform decisions are grounded in workflow needs.

Acceptance criteria:

- Functional requirements are documented.
- Non-functional requirements are documented.
- Platform decision is deferred until workflow is clear.

### Story 5.2: Candidate platform options

As an architect, I want Microsoft, Google, Amazon, GitHub-first, and hybrid paths compared so that we can choose intelligently.

Acceptance criteria:

- Candidate paths are documented.
- Strengths are noted.
- First recommendation is platform-light.

### Story 5.3: Architecture decision log

As an architect, I want an architecture decision log so that major decisions are recorded.

Acceptance criteria:

- ADR template exists.
- First decisions are logged.
- Future decisions have a clear place to go.

## Epic 6: Delivery Management

### Goal

Manage the work through releases, sprints, and checkpoints.

### Story 6.1: Release roadmap

As product lead, I want a release roadmap so that the product progresses in controlled stages.

Acceptance criteria:

- Releases are defined.
- Each release has purpose, outputs, and success criteria.
- Roadmap prevents overbuilding.

### Story 6.2: Sprint plan

As delivery lead, I want a sprint plan so that the next work can be executed in order.

Acceptance criteria:

- Sprint 1 is defined.
- Sprint 2 is defined.
- Sprint outcomes are clear.

### Story 6.3: Definition of ready and done

As delivery lead, I want ready and done criteria so that work does not drift.

Acceptance criteria:

- Definition of ready exists.
- Definition of done exists.
- Work items can be assessed consistently.

## Priority order

1. Pilot foundation
2. Scenario and prompt workflow
3. User experience
4. Evidence and validation
5. Delivery management
6. Architecture and platform

## Backlog conclusion

The backlog is focused on pilot readiness and controlled productisation.

The first build should prove the loop before scaling the platform.
