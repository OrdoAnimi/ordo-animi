# VALOUR Pilot Workflow

Version: 0.1
Status: Private pilot workflow
Release: Pilot Ready

## Purpose

This document defines the operational workflow for running the first VALOUR pilot.

The workflow is designed to work manually first. It can later become the basis for software screens, agent flows, automation, and cloud architecture.

## Workflow summary

The VALOUR pilot moves a user through nine stages:

1. Intake
2. Scenario selection
3. Situation framing
4. Preparation
5. Rehearsal
6. Language refinement
7. Real-world application
8. After-action review
9. Pattern report and feedback capture

## Stage 1: Intake

### Objective

Understand the user's role, context, current leadership challenge, and upcoming architecture leadership situation.

### Inputs

- User role
- Organisation context
- Architecture background
- Current leadership challenge
- Upcoming conversation or meeting
- Confidence baseline
- Desired outcome

### Output

A completed intake record.

### Operator action

Review the intake and identify the most suitable VALOUR scenario.

## Stage 2: Scenario selection

### Objective

Match the user to the right pilot scenario.

### Inputs

- Intake form
- User's upcoming situation
- Scenario catalogue

### Output

Selected scenario and pilot focus.

### Operator action

Confirm whether the situation fits an existing scenario or requires a variant.

## Stage 3: Situation framing

### Objective

Clarify the context before preparation begins.

### Inputs

- Audience
- Decision required
- Stakeholders
- Risk
- Desired outcome
- What could go wrong

### Output

A short situation frame that can be used in prompts and preparation notes.

### Operator action

Reduce the situation into plain language.

## Stage 4: Preparation

### Objective

Prepare the user before rehearsal.

### Inputs

- Situation frame
- Selected scenario
- Audience needs
- Decision required
- Likely questions

### Output

A preparation brief.

The brief should include:

- Recommended opening frame
- Key message
- Questions to ask
- Likely pushback
- Risks to avoid
- Closing statement

## Stage 5: Rehearsal

### Objective

Allow the user to practise the difficult part before the real moment.

### Inputs

- Preparation brief
- Scenario prompt
- User response

### Output

Rehearsal transcript or notes.

### Operator action

Run the scenario prompt and capture the user's answers and AI feedback.

## Stage 6: Language refinement

### Objective

Improve the user's language so it is clearer, shorter, and more appropriate for the audience.

### Inputs

- User's original answer
- Rehearsal feedback
- Scenario context

### Output

Improved language options.

The output should include:

- Short version
- Executive version
- Empathetic version
- Decision version

## Stage 7: Real-world application

### Objective

The user applies the preparation in the actual meeting or conversation.

### Inputs

- Preparation brief
- Refined language
- User judgement

### Output

Real-world result.

### Operator action

Do nothing during the meeting unless a future version supports live assistance. The first pilot is preparation and review only.

## Stage 8: After-action review

### Objective

Capture what happened and what can be learned.

### Inputs

- User summary of the real conversation
- Outcome
- Stakeholder response
- What worked
- What did not work

### Output

After-action review notes.

The review should identify:

- Clarity
- Cadence
- Empathy
- Authority
- Decision framing
- Over-explaining
- Missed opportunities
- Next improvement

## Stage 9: Pattern report and feedback capture

### Objective

Turn the pilot into evidence and development insight.

### Inputs

- Intake
- Preparation brief
- Rehearsal notes
- Language refinement
- After-action review
- User feedback

### Output

Pilot completion record.

The record should include:

- Starting challenge
- Scenario used
- Improvement observed
- User feedback
- Pattern detected
- Recommended next focus
- Testimonial candidate if appropriate

## Workflow completion criteria

A pilot cycle is complete when the user has moved through intake, scenario selection, preparation, rehearsal, language refinement, real-world use, after-action review, and feedback capture.

## Workflow principle

Do not overcomplicate the first pilot.

One user. One real scenario. One rehearsal. One review. One improvement pattern.
