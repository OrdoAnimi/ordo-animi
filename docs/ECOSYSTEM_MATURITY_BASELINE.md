## Operating Principle

> **AI drafts. Human reviews. PMO governs. Client receives controlled artefacts.**

No AI-generated artefact goes to a client without human review, PMO approval, and lifecycle governance.

---

## Shared Artefact Lifecycle

```
REQUIRED → DRAFT → IN REVIEW → APPROVED → SHARED → LOCKED → HANDOVER
                       │
                   REJECTED → DRAFT
```

| State | Meaning |
|-------|---------|
| REQUIRED | Artefact identified as needed, not yet generated |
| DRAFT | Generated by a limb generator — not yet reviewed |
| IN REVIEW | Assigned to a human reviewer |
| APPROVED | Reviewed and approved by an accountable practitioner |
| SHARED | Made visible to client or stakeholder (controlled) |
| LOCKED | Frozen — no further edits permitted |
| HANDOVER | Formally handed over to client or approved storage |

All limb generators produce artefacts at DRAFT state. PMO Portal controls the remainder of the lifecycle.

## EA Artefact Generator as Maturity Reference

The EA Artefact Generator is the **maturity reference baseline** for all limb generators. When assessing whether a limb is at sufficient practical maturity, measure it against EA:

- Product intent is clearly stated
- Who it is for is documented
- What it generates or governs is catalogued
- A live or deployable route exists
- Ecosystem fit (input, output, PMO handoff, Ordo governance) is documented
- Run/build/deploy instructions are complete
- Security and data handling notes exist
- A roadmap exists
- Licence posture is stated

## Minimum Maturity Standard for All Limbs

Each limb generator must meet this minimum standard before being considered production-ready:

### 1. Product Intent
A clear statement of what the tool does, who it is for, and what professional problem it solves.

### 2. Live / Deployable Route
Either a live URL or a committed deployment configuration that can be executed without further code changes.

### 3. VAF Ecosystem Fit
Documentation of:
- What inputs the limb receives (from practitioners, Ordo, EA, BA)
- What artefacts it generates
- How it hands off to PMO Portal
- How Ordo Animi governs or orchestrates it

### 4. Artefact Catalogue
A list of every artefact type the limb generates or governs, with enough description to know what each is for.

### 5. Generation or Preview Mode
A working UI or API path that produces an artefact draft from context input.

### 6. Output Management
Ability to view, copy, download, or export generated artefacts.

### 7. Export Path
At least one export format (Markdown, JSON, CSV) for getting artefacts out of the browser and into governed storage.

### 8. Lifecycle Language
The shared VAF lifecycle states (REQUIRED → DRAFT → IN REVIEW → APPROVED → SHARED → LOCKED → HANDOVER) are understood and referenced.

### 9. Security and Data Notes
A clear statement that:
- Prototype environments must not receive real confidential client data
- Generated outputs are drafts until reviewed
- API keys are never in browser code
- Client-ready outputs go to private approved storage

### 10. Run / Build / Deploy Instructions
Complete commands for local development, production build, and deployment. No ambiguity.

### 11. Roadmap
A brief statement of what comes next: generation improvements, persistence, PMO handoff, export automation, Ordo integration.

### 12. Licence Posture
Private proprietary, or clearly stated terms. No accidental open-source exposure.

## Release Gate Questions

Before a limb is considered ready for production or client engagement use, answer these questions:

1. **What professional work does this repo own?**
   What discipline, practice area, or role does this tool serve?

2. **What artefacts does it generate or govern?**
   Name every artefact type. Be specific.

3. **What lifecycle states does it support?**
   Which states can a practitioner move an artefact through within this tool?

4. **How does it hand off to PMO Portal?**
   Export format, handoff mechanism (JSON, API, manual), and which fields map to the PMO handoff contract.

5. **How does Ordo Animi orchestrate or govern it?**
   What decisions, guardrails, or context does Ordo provide to this limb?

6. **What data is safe to enter?**
   Synthetic data? Anonymised? Real client data under what conditions?

7. **How is output exported, reviewed, approved, shared, and locked?**
   Describe the full output path from generation to client delivery.
