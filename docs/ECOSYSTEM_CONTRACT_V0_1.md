# Velocity Architecture Ecosystem — Shared Contract V0.1

Version: V0.1 | Date: 2026-06-05 | Status: Logical contract — no API implementation required

---

## Purpose

This document defines the shared logical contract between limb generators (EA, BA, SA, PM) and PMO Portal for artefact handoff and lifecycle governance. It is not an API specification. It is the agreed data model that governs what information moves between systems and what PMO Portal expects to receive.

---

## Contract Principle

1. Limb generators produce artefacts in **DRAFT** state.
2. Artefacts are handed to **PMO Portal** for lifecycle governance.
3. **Ordo Animi** provides decision context and guardrails — it does not generate artefacts.
4. **Human approval** is required before any artefact moves past APPROVED state.
5. No client-visible artefact is released without PMO Portal lifecycle control.

## Artefact Request Object

A practitioner or Ordo Animi sends a request to a limb generator:

```json
{
  "requestId": "uuid",
  "sourceGenerator": "EA | BA | SA | PM",
  "artefactType": "string — e.g. 'Architecture Decision Record'",
  "artefactTitle": "string",
  "engagementContext": {
    "organisation": "string",
    "initiative": "string",
    "outcomes": "string",
    "stakeholders": "string",
    "assumptions": "string",
    "risks": "string",
    "constraints": "string"
  },
  "requestedBy": "string — practitioner identifier",
  "requestedAt": "ISO8601 timestamp",
  "lifecycleState": "REQUIRED"
}
```

## Artefact Response Object

The limb generator returns:

```json
{
  "artefactId": "uuid",
  "sourceGenerator": "EA | BA | SA | PM",
  "artefactType": "string",
  "artefactTitle": "string",
  "markdownContent": "string — full artefact content in Markdown",
  "structuredObjects": "object | null — JSON representation if available",
  "exportFormats": ["markdown", "json", "csv"],
  "generatedAt": "ISO8601 timestamp",
  "modelUsed": "string | null — AI model identifier if AI-generated",
  "lifecycleState": "DRAFT"
}
```

## Governance Metadata

Assigned by the practitioner or PMO Portal on receipt:

```json
{
  "governanceOwner": "string — engagement lead or delivery owner",
  "reviewOwner": "string — practitioner assigned to review",
  "approvalState": "PENDING_REVIEW | IN_REVIEW | APPROVED | REJECTED | LOCKED",
  "approvedBy": "string | null",
  "approvedAt": "ISO8601 timestamp | null",
  "reviewNotes": "string | null"
}
```

## Handoff Metadata

Governs the move from PMO Portal to client delivery or storage:

```json
{
  "targetSystem": "string | null — storage system or client workspace",
  "handoffState": "NOT_STARTED | IN_PROGRESS | COMPLETE",
  "handoffAt": "ISO8601 timestamp | null",
  "storageTarget": "string | null — specific storage path or reference",
  "clientVisible": "boolean — whether client can currently see this artefact",
  "pmoLifecycleState": "REQUIRED | DRAFT | IN_REVIEW | APPROVED | SHARED | LOCKED | HANDOVER"
}
```

## Human Approval State

```
PENDING_REVIEW  — generated, not yet assigned for review
IN_REVIEW       — assigned to a reviewer
APPROVED        — reviewed and approved by an accountable practitioner
REJECTED        — returned for rework → reverts to DRAFT
LOCKED          — approved and frozen, no further changes
```

Transition rules:
- Only a named human practitioner can move an artefact to APPROVED
- LOCKED can only be set after APPROVED
- REJECTED always returns the artefact to DRAFT

---

## PMO Portal Lifecycle State

```
REQUIRED → DRAFT → IN_REVIEW → APPROVED → SHARED → LOCKED → HANDOVER
                       ↓
                   REJECTED → DRAFT
```

| State | Description |
|-------|-------------|
| REQUIRED | Artefact needed, not yet generated |
| DRAFT | Generated by a limb generator |
| IN_REVIEW | Assigned for human review |
| APPROVED | Reviewed and approved |
| SHARED | Made visible to client (controlled) |
| LOCKED | Frozen — no edits |
| HANDOVER | Formally handed to client or approved storage |

## Implementation Notes

**V0.1 is a logical contract only.** No API implementation is required today.

Current interim path:
1. Practitioner generates artefact in limb generator
2. Exports as JSON or Markdown
3. Manually imports to PMO Portal
4. PMO Portal assigns lifecycle state and review owner
5. Practitioner reviews and approves within PMO Portal
6. PMO Portal controls client visibility and handover

**Future path (V1.0+):**
- Direct API between limb generators and PMO Portal
- Ordo Animi orchestrates requests and routes to correct limb
- PMO Portal receives artefacts via webhook or API call
- Real-time lifecycle state sync
