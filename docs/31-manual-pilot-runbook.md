# VALOUR Manual Pilot Runbook

Version: 0.1
Status: Private runbook
Release: Pilot Ready

## Purpose

This runbook explains how to run the first VALOUR pilot manually before a software platform exists.

The runbook is written for the founder or pilot operator.

## Pilot operating rule

One user.

One real architecture leadership scenario.

One preparation cycle.

One rehearsal cycle.

One after-action review.

One pattern report.

Keep the loop simple until it proves value.

## Before the pilot

### Prepare materials

Confirm the following documents are available:

- Pilot pack
- Intake form
- Scenario library
- Scenario prompt pack
- Pilot templates
- Data model
- Validation tracker

### Create pilot folder

Create a private folder for the user under `pilot-data/`.

Suggested structure:

`pilot-data/users/user-001/`

Inside the folder, create:

- `01-intake.md`
- `02-scenario.md`
- `03-preparation.md`
- `04-rehearsal.md`
- `05-language-refinement.md`
- `06-after-action-review.md`
- `07-pattern-report.md`

## Step 1: Intake

Ask the user to complete the intake form.

Review the intake and identify:

- Role
- Context
- Upcoming leadership situation
- Desired outcome
- Main risk
- Confidence baseline
- Likely scenario fit

Output: completed intake record.

## Step 2: Scenario selection

Match the user situation to the closest scenario.

If the situation does not exactly match an existing scenario, choose the nearest scenario and note the variant.

Output: selected scenario and scenario variant.

## Step 3: Situation framing

Write a short situation frame.

The frame should answer:

- Who is the audience?
- What decision or outcome is required?
- What is at stake?
- What could go wrong?
- What must the user communicate clearly?

Output: situation frame.

## Step 4: Preparation brief

Use the preparation prompt for the selected scenario.

Create a preparation brief containing:

- Opening frame
- Key message
- Decision required
- Questions to ask
- Likely pushback
- Risks to avoid
- Recommended tone
- Closing statement

Output: preparation brief.

## Step 5: Rehearsal

Use the scenario rehearsal prompt.

Run one rehearsal sequence.

Capture:

- Prompt used
- User answer
- Feedback
- Improved answer
- Notes on clarity, cadence, empathy, authority, and decision framing

Output: rehearsal record.

## Step 6: Language refinement

Take the user's weakest or most important answer and improve it.

Create four versions:

- Short version
- Executive version
- Empathetic version
- Decision version

Output: language refinement record.

## Step 7: Real-world use

The user applies the preparation in the real meeting or conversation.

No live support is required for the first pilot.

Output: real-world event completed.

## Step 8: After-action review

After the real event, run the after-action review.

Capture:

- What happened
- What worked
- What did not work
- Stakeholder response
- Outcome
- Confidence after support
- What should change next time

Output: after-action review record.

## Step 9: Pattern report

Prepare a short pattern report for the user.

The report should include:

- Starting challenge
- Scenario used
- Main improvement
- Primary pattern detected
- Primary strength
- Primary risk
- Recommended next focus
- Pilot usefulness score
- Recommended next step

Output: pattern report.

## Step 10: Validation update

Update the validation tracker.

Capture:

- Whether the user found it useful
- Whether they would continue
- Whether they would recommend it
- Whether they would pay or sponsor
- Which scenario was most valuable
- What should change before the next pilot

Output: validation tracker updated.

## Runbook completion criteria

The pilot is complete when the user has received a pattern report and feedback has been captured.

## Operator discipline

Do not turn the pilot into a general coaching session.

Do not over-explain Ordo Animi.

Do not solve every personal issue.

Do not collect confidential architecture material unnecessarily.

Do not build software during the pilot.

Use the pilot to test the behaviour loop.
