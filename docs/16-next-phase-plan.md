# Next Phase Plan

Version: 0.1
Status: Private phase plan
Phase: Pilot Readiness

## Purpose

This document defines the next phase after the Foundation Release.

The Foundation Release created the product source of truth. The Pilot Readiness phase turns that documentation into a usable pilot package that can be shown to early users, advisors, and funders.

## Phase objective

Move VALOUR from internal product definition to first external validation.

The goal is not to build a full platform. The goal is to prepare the first pilot so it can be tested with real architecture leaders.

## Phase principle

No expansion without validation.

The next phase should not add more product branches, more doctrine, or more speculative platform features unless they directly support the first VALOUR pilot.

## Current baseline

The repo now contains:

- Product charter
- Founder story
- System architecture
- Market positioning
- Go-to-market strategy
- Funder engagement pack
- MVP plan
- Scenario library
- Pilot templates
- Outreach copy
- Pitch narrative
- Product roadmap
- Operating model
- Risk and IP notes
- Funding one-pager
- Release summary

## Next phase deliverables

### 1. Pilot pack

Create a single pilot pack that can be shared with selected users or advisors.

The pilot pack should explain:

- What VALOUR is
- Who it is for
- What the pilot includes
- What the user needs to bring
- What the user gets at the end
- How feedback will be captured

### 2. Pilot intake form

Create a structured intake form for pilot users.

It should capture role, context, upcoming leadership scenario, communication challenge, confidence level, and desired outcome.

### 3. Scenario prompt pack

Turn the scenario library into reusable AI prompts.

Each scenario should have a preparation prompt, rehearsal prompt, language refinement prompt, and after-action review prompt.

### 4. Landing page draft

Create a private landing page draft for VALOUR.

This can later become a simple public page, but for now it should clarify message, audience, offer, and call to action.

### 5. Founder pitch script

Create a short spoken pitch for funder and advisor conversations.

It should be clear enough to say in under two minutes.

### 6. Discovery call script

Create a short discovery call flow for architecture leaders and potential pilot users.

The purpose is to validate pain before selling too hard.

### 7. Validation tracker

Create a simple tracker for outreach, calls, pilot candidates, feedback, and next actions.

This can be a Markdown table first.

### 8. Evidence archive structure

Create archive folders for origin evidence, pilot notes, feedback, and future research.

Do not upload sensitive external evidence until it is organised and reviewed.

## Pilot Readiness success criteria

The phase is complete when:

- A pilot pack exists.
- A pilot intake form exists.
- Scenario prompts are ready.
- Outreach copy is ready.
- A short pitch script exists.
- A discovery script exists.
- A validation tracker exists.
- The first ten target contacts can be approached.

## First external validation target

The first validation target is to speak with five architecture or technical leadership contacts.

The goal is to test:

- Is the problem real?
- Is the language clear?
- Would they use it?
- Would they pay or sponsor a user?
- Which scenario is most valuable?
- What is missing from the offer?

## Decision gate after Pilot Readiness

After Pilot Readiness, the next decision is whether to run:

- Founder-led individual pilot
- Small team pilot
- Advisor-only feedback round
- Funder engagement round
- Landing page pre-launch

## What to avoid in this phase

Do not build the full desktop agent yet.

Do not create a public product launch yet.

Do not widen into generic leadership yet.

Do not lead with VITA as a wellness product yet.

Do not over-explain Ordo Animi to early users.

Do not give full architecture access to advisors unless there is a clear reason.

## Phase summary

Foundation defined the product.

Pilot Readiness prepares the product for contact with the market.

The next proof is not theoretical. The next proof is whether real architecture leaders recognise the problem and want the pilot.
