# VALOUR™ MVP Plan

Version: 0.1
Status: Private MVP plan

## Purpose

This document defines the first minimum viable product for VALOUR for Architecture Leadership.

The MVP is designed to validate the method before building a full software platform.

## MVP thesis

If architects can use VALOUR to prepare for, rehearse, and review real leadership conversations, then the product has commercial potential.

The MVP must prove behavioural usefulness before technical scale.

## What the MVP is

The MVP is a founder-led, AI-assisted pilot.

It includes:

- Scenario templates
- Meeting preparation prompts
- AI rehearsal prompts
- After-action review prompts
- User feedback capture
- Pattern review
- Basic pilot documentation

## What the MVP is not

The MVP is not a full SaaS product.

It is not a finished desktop agent.

It is not a public Ordo Animi platform.

It is not a consumer wellness product.

It is not a generic leadership marketplace.

## Core user loop

1. User selects a real upcoming leadership situation.
2. VALOUR helps clarify audience, context, decision, risk, and desired outcome.
3. User rehearses the situation through AI role-play.
4. VALOUR helps refine language and cadence.
5. User enters the real situation.
6. User completes an after-action review.
7. VALOUR identifies patterns and next improvement.

## First scenario set

### 1. Architecture review board

User needs to present a decision, design, risk, or recommendation to an architecture review forum.

### 2. Executive briefing

User needs to explain a complex technology decision to senior stakeholders.

### 3. Discovery workshop

User needs to guide a group through ambiguity and extract useful information.

### 4. Challenging a poor solution

User needs to challenge a solution without becoming dismissive or obstructive.

### 5. Security or risk pushback

User needs to handle challenge from security, risk, compliance, or governance stakeholders.

### 6. Delivery conflict

User needs to work through conflict with a project manager, delivery lead, vendor, or implementation team.

### 7. Architecture trade-off explanation

User needs to explain options, trade-offs, constraints, risks, and decision criteria.

### 8. Strategy-to-architecture translation

User needs to turn strategic intent into architecture options and next steps.

### 9. First meeting with a new team

User needs to build trust, establish role clarity, and avoid over-talking or over-positioning.

### 10. End-of-day handover

User closes the professional loop and hands over to the personal VITA layer for recovery and life rhythm.

## Pilot delivery model

The first pilot can be delivered manually with AI support.

Recommended pilot flow:

- Intake form
- Scenario selection
- Weekly check-in
- AI rehearsal session
- Real-world application
- After-action review
- Final pattern report

## Outputs for each user

Each user should finish with:

- A leadership pattern snapshot
- Scenario preparation notes
- Improved language patterns
- Meeting rehearsal history
- After-action notes
- Recommended next development focus

## Success criteria

The MVP is successful if users report that VALOUR helped them:

- Prepare more clearly
- Speak with better cadence
- Reduce over-explaining
- Handle challenge more calmly
- Show empathy without losing authority
- Improve stakeholder trust
- Review performance more honestly

## Measurement

Initial measurement can be simple and qualitative.

Before scenario:

- Confidence level
- Clarity level
- Perceived risk
- Desired outcome

After scenario:

- What happened
- What worked
- What failed
- What language helped
- What would change next time
- Whether VALOUR helped

## Build sequence

### Release 0: Manual founder protocol

Use ChatGPT and written prompts to run the process manually with the founder as operator.

### Release 1: Prompt pack and scenario library

Create reusable scenario prompts and templates.

### Release 2: Private pilot

Recruit pilot users and run the process manually.

### Release 3: Lightweight interface

Build a simple front end only after the pilot proves the loop.

### Release 4: Integrated agent

Explore calendar, meeting, notes, and mode-switch integrations after evidence exists.

## MVP discipline

Do not build the full Jarvis-style product first.

Do not expose the whole Ordo Animi doctrine to early users.

Do not ask users to understand the full ecosystem.

Help them with one real leadership moment.

If that works, expand.
