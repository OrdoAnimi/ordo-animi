# Enterprise Architecture Brief

Version: 0.1
Status: Private architecture brief
Phase: Architecture and Pilot Build

## Purpose

This document defines the enterprise architecture view for Ordo Animi and VALOUR.

The enterprise architecture view explains how the pilot product fits into the broader ecosystem, what capabilities are required, and how the system should evolve without losing the first commercial focus.

## Enterprise context

Ordo Animi is the parent system.

VALOUR is the first commercial product.

Velocity Architecture is the first professional domain context.

The first customer pathway is architecture and technical leadership.

VITA, VALLUM, VERITAS, and VIA remain part of the broader ecosystem but are not all shipped as public products in the first release.

## Enterprise architecture intent

The architecture intent is to create a controlled agentic AI product ecosystem that can begin as a pilot workflow and later expand into a multi-agent operating system.

The design must support:

- Pilot delivery
- Scenario-based AI guidance
- User preparation
- Rehearsal
- Review
- Evidence capture
- Pattern reporting
- Future personal mode handover
- Future enterprise deployment

## Strategic principles

### Start with the wedge

VALOUR for Architecture Leadership is the first product and must remain the first commercial focus.

### Prove before scaling

Do not build a large platform before proving the workflow with real users.

### Keep the parent architecture intact

The broader Ordo Animi model should inform the design, but it should not confuse the first product.

### Support cloud portability

The founder has Microsoft, Google, and Amazon partner context. The early design should avoid unnecessary lock-in until the customer route is clear.

### Keep human judgement central

VALOUR helps users prepare, rehearse, and review. It does not replace professional judgement.

## Enterprise capability domains

### Product capability

Defines VALOUR offer, pilot flow, scenarios, pricing path, and user experience.

### AI capability

Defines prompts, agent roles, orchestration, model selection, review logic, and quality controls.

### Data capability

Defines user records, scenario records, rehearsal records, after-action reviews, pattern reports, and evidence capture.

### Security and privacy capability

Defines access control, data minimisation, confidentiality, retention, and safe handling of pilot records.

### Delivery capability

Defines releases, MVP, MLP, epics, stories, sprints, acceptance criteria, and deployment pathway.

### Commercial capability

Defines pilot offer, user outreach, funder materials, partner path, and future enterprise sale.

### Operating capability

Defines how the founder or operator runs the manual pilot before the product is automated.

## Target state vision

The long-term target state is an agentic operating system that can support professional and personal modes.

For the first product, the target state is narrower:

A user brings a real architecture leadership situation. VALOUR guides preparation, rehearsal, language refinement, real-world review, and pattern reporting.

## Current state

The current state is a private GitHub product workspace containing product, pilot, architecture, GTM, and release documentation.

There is no deployed platform yet.

The pilot can run manually using documents and prompts.

## Transition state

The next transition is from document-based pilot to lightweight operating workflow.

This can begin manually and later move into a simple interface or agent workflow.

## Future state

Future states may include:

- Private pilot portal
- User intake interface
- Scenario selection interface
- AI rehearsal workspace
- Pattern report generation
- Team pilot reporting
- Calendar integration
- VALLUM work-to-life handover
- VITA personal mode support
- Enterprise deployment options

## Architecture governance

Architecture decisions should be recorded in an architecture decision log.

Key decisions include:

- First implementation stack
- Data storage model
- AI model/provider choice
- Prompt orchestration pattern
- Privacy and retention model
- Manual versus automated workflow boundaries
- Enterprise integration roadmap

## First architecture decision

The first decision is not cloud platform.

The first decision is pilot operating pattern:

Manual workflow, lightweight interface, or agent-assisted workflow.

The recommended path is manual workflow first, then lightweight interface, then agent-assisted workflow.

## Enterprise architecture conclusion

The enterprise architecture supports a staged path from founder-led pilot to agentic product.

VALOUR remains the first wedge. Ordo Animi remains the parent ecosystem. Architecture leadership remains the first market.
