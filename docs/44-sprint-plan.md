# VALOUR Sprint Plan

Version: 0.1
Status: Private sprint plan
Phase: Architecture and Pilot Build

## Purpose

This document defines the first sprint plan for VALOUR for Architecture Leadership.

The sprint plan turns the architecture and backlog into an executable delivery sequence.

## Delivery approach

VALOUR will use short focused sprints with clear release gates.

Each sprint should produce usable product or architecture artefacts, not loose thinking.

## Sprint principles

Keep the product focused on architecture leadership.

Protect the VALOUR loop: prepare, rehearse, refine, apply, review, report, learn.

Use documentation first where it accelerates clarity.

Move to build only when the workflow is stable enough.

Avoid platform overbuild before pilot evidence.

## Sprint 0: Architecture setup

### Goal

Create the architecture and delivery foundation for the pilot build.

### Deliverables

- Agentic AI architecture program
- Enterprise architecture brief
- Business architecture brief
- Solution brief
- Capability map
- Value stream map
- MVP and MLP definition
- Release roadmap
- Epic and story backlog
- Sprint plan
- Diagram set
- Architecture decision log

### Exit criteria

The project has enough structure to move into build planning without re-explaining the concept.

## Sprint 1: Manual pilot operating system

### Goal

Create the complete manual pilot operating system.

### Stories

- Create standard pilot-data folder structure.
- Create pilot user folder template.
- Create scenario record template.
- Create preparation record template.
- Create rehearsal record template.
- Create review record template.
- Create pattern report template.
- Create validation record template.

### Outputs

- Pilot folder structure
- Reusable templates
- Operator-ready runbook

### Exit criteria

One pilot user can be onboarded and run through the full manual flow.

## Sprint 2: Scenario workflow hardening

### Goal

Make the scenario and prompt workflow reliable enough for repeated use.

### Stories

- Refine scenario selection criteria.
- Add scenario-to-prompt mapping.
- Add scenario variants.
- Add output quality criteria.
- Add common failure patterns.
- Add operator notes for each scenario.

### Outputs

- Hardened scenario workflow
- Prompt operating guide
- Scenario selection matrix

### Exit criteria

The operator can select the right scenario and run the correct prompt path without improvising.

## Sprint 3: Pilot experience pack

### Goal

Prepare the user-facing pilot experience.

### Stories

- Finalise pilot pack.
- Finalise intake form.
- Finalise landing page draft.
- Finalise founder script.
- Finalise discovery notes.
- Create pilot invitation pack.
- Create follow-up email pack.

### Outputs

- User-facing pilot materials
- Outreach-ready material
- Pilot invitation workflow

### Exit criteria

The founder can approach a pilot user and explain the offer clearly.

## Sprint 4: Architecture decision sprint

### Goal

Prepare the first technical architecture decision.

### Stories

- Create architecture decision log.
- Create ADR template.
- Create Microsoft option brief.
- Create Google option brief.
- Create Amazon option brief.
- Create hybrid low-cost option brief.
- Compare options against pilot workflow.

### Outputs

- Architecture option pack
- ADRs
- Recommended first build path

### Exit criteria

The first implementation architecture can be selected without re-opening the entire product strategy.

## Sprint 5: Lightweight pilot build brief

### Goal

Create the build brief for the first lightweight implementation.

### Stories

- Define functional requirements.
- Define non-functional requirements.
- Define user screens or pages.
- Define data objects.
- Define prompt orchestration flow.
- Define report generation flow.
- Define deployment and operating model.

### Outputs

- Build brief
- UX flow
- Data flow
- Agent flow
- Deployment notes

### Exit criteria

A developer or AI coding tool can begin implementation with minimal clarification.

## Sprint 6: Pilot execution

### Goal

Run the first real or simulated pilot.

### Stories

- Create pilot user record.
- Select scenario.
- Run preparation.
- Run rehearsal.
- Refine language.
- Complete review.
- Generate pattern report.
- Capture evidence.

### Outputs

- Completed pilot record
- Pattern report
- Evidence record
- Improvement backlog

### Exit criteria

One pilot loop is completed end to end.

## Sprint governance

Each sprint should end with:

- Completed outputs
- Open issues
- Product risks
- Architecture risks
- Next sprint decision

## Definition of ready

A work item is ready when the purpose, inputs, expected output, and acceptance criteria are clear.

## Definition of done

A work item is done when the artefact exists in the repo, is readable, supports the next release, and does not create unresolved product ambiguity.
