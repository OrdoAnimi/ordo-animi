# Pattern Report

Pilot: Pilot 001 - Architecture Review Board

Status: Complete

## Leadership Pattern

The user's core pattern is technical over-explanation under pressure.

When the room asks for decision clarity, the user tends to add more architecture detail. This is understandable because the user is technically strong, but it creates leadership risk in governance settings.

## Primary Strength

The user has strong technical judgement and can see design options, constraints, controls, and delivery implications clearly.

This gives the user credibility, but only if the judgement is translated into decision language.

## Primary Risk

The primary risk is losing executive attention and sounding defensive under challenge.

The board may interpret long technical answers as uncertainty, even when the user is confident.

## Best Scenario Type

Best fit:

Architecture governance conversations where the user has enough technical knowledge but needs to improve framing, cadence, trade-off language, and close.

Examples:

- architecture review board
- security design review
- platform exception review
- delivery risk checkpoint

## Hardest Scenario Type

Hardest fit:

Ambiguous stakeholder conflict where the user must influence without clear decision authority.

Reason:

The user is most comfortable when the technical problem is clear. They may find it harder when the central issue is political alignment, sponsorship, or competing stakeholder incentives.

## Development Focus

Primary development focus:

Answer discipline.

Specific behaviours:

- state the decision first
- give one recommendation before explaining options
- name the trade-off directly
- answer the question asked
- stop before adding unnecessary detail
- close with the next action

## Coaching Note

"Your technical depth is the asset; your next step is to make the board feel the decision is simple enough to make."
